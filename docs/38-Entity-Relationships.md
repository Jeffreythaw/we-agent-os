# Entity Relationships

This document defines the core entities in WE Agent OS. For each, we define Purpose, Key fields, Ownership, Relationships, Lifecycle, Validation rules, and Risks.

## 1. Workspace
- **Purpose**: Top-level isolation boundary for tenant data.
- **Key fields**: `id`, `name`, `createdAt`
- **Ownership**: Root
- **Relationships**: Contains Projects, Skills, Workflows, Policies.
- **Lifecycle**: Created -> Active -> Archived
- **Validation rules**: Must have a unique ID and name.
- **Risks**: Data leakage if boundaries are breached.

## 2. Project
- **Purpose**: The site/client/job boundary. Groups related work together.
- **Key fields**: `id`, `workspaceId`, `name`, `status`
- **Ownership**: Workspace
- **Relationships**: Contains Personas, Knowledge, Artifacts, Executions.
- **Lifecycle**: Created -> Active -> Completed -> Archived
- **Validation rules**: Must belong to a valid Workspace.
- **Risks**: Overloading a single project with unrelated data.

## 3. Persona
- **Purpose**: Role identity, tone, and expertise configuration. Not a runtime execution.
- **Key fields**: `id`, `projectId`, `name`, `promptTemplate`
- **Ownership**: Project
- **Relationships**: Adopted by Agents during Execution.
- **Lifecycle**: Draft -> Active -> Deprecated
- **Validation rules**: Must have a clear identity definition.
- **Risks**: Confusing persona configuration with runtime state.

## 4. Agent
- **Purpose**: The runtime execution process that adopts a Persona.
- **Key fields**: `id`, `executionId`, `status`
- **Ownership**: Execution / Kernel
- **Relationships**: Adopts 1 Persona, executes Tasks.
- **Lifecycle**: Initializing -> Running -> Suspended -> Completed/Failed
- **Validation rules**: Cannot exist without an Execution context.
- **Risks**: Zombie processes if lifecycle is not managed.

## 5. Skill
- **Purpose**: Reusable atomic building block (Standard Operating Procedure).
- **Key fields**: `id`, `name`, `schema`
- **Ownership**: Workspace
- **Relationships**: Composed by Workflows, utilized by Agents.
- **Lifecycle**: Draft -> Active -> Deprecated
- **Validation rules**: Must declare its required inputs and tools.
- **Risks**: Creating monolithic skills instead of atomic ones.

## 6. Workflow
- **Purpose**: A graph composing multiple Skills into a larger process.
- **Key fields**: `id`, `name`, `steps`
- **Ownership**: Workspace
- **Relationships**: Composes multiple Skills.
- **Lifecycle**: Draft -> Active -> Deprecated
- **Validation rules**: Cannot contain circular dependencies unless explicitly typed as iterative loops.
- **Risks**: Infinite loops in poor graph designs.

## 7. Task
- **Purpose**: A specific instance of work generated by the Planner.
- **Key fields**: `id`, `workflowId`, `status`
- **Ownership**: Execution
- **Relationships**: Mapped to Skills.
- **Lifecycle**: Pending -> Running -> Completed/Failed
- **Validation rules**: Must have clear success criteria.
- **Risks**: Ambiguous task definitions causing LLM hallucinations.

## 8. Execution
- **Purpose**: The immutable runtime history of a process.
- **Key fields**: `id`, `projectId`, `status`, `startTime`, `endTime`
- **Ownership**: Project
- **Relationships**: Owns Tasks, Agents, and resulting Artifacts.
- **Lifecycle**: Started -> Running -> Finished
- **Validation rules**: History is append-only.
- **Risks**: unbounded execution loops consuming excessive resources.

## 9. Tool
- **Purpose**: Device drivers providing system/world access.
- **Key fields**: `name`, `riskLevel`
- **Ownership**: Workspace
- **Relationships**: Requested by Skills, authorized by Policy Engine.
- **Lifecycle**: Registered -> Active -> Unregistered
- **Validation rules**: Must have a strict input schema.
- **Risks**: Broadly scoped tools (e.g., raw bash execution).

## 10. Provider
- **Purpose**: Connection to specific LLM endpoints (e.g., OpenAI, Ollama).
- **Key fields**: `id`, `type`, `config`
- **Ownership**: Workspace
- **Relationships**: Selected by Reasoner.
- **Lifecycle**: Configured -> Active -> Disabled
- **Validation rules**: Secrets must not be stored in plaintext.
- **Risks**: Hardcoded API keys.

## 11. Memory
- **Purpose**: Operational and experiential context for active processes.
- **Key fields**: `id`, `contextId`, `data`
- **Ownership**: Execution (Short-term) / Project (Long-term)
- **Relationships**: Passed to Agents.
- **Lifecycle**: Initialized -> Appended -> Flushed/Archived
- **Validation rules**: Must be strictly size-managed to fit context windows.
- **Risks**: Context overflow leading to dropped information.

## 12. Policy
- **Purpose**: Rules evaluated before risky actions.
- **Key fields**: `id`, `resource`, `action`, `effect`
- **Ownership**: Workspace
- **Relationships**: Enforced by Policy Engine.
- **Lifecycle**: Active
- **Validation rules**: Default to deny.
- **Risks**: Overly permissive wildcard policies.

## 13. Approval
- **Purpose**: Explicit user decisions on pending risky actions.
- **Key fields**: `id`, `status`
- **Ownership**: Execution
- **Relationships**: Tied to a specific Task and Policy constraint.
- **Lifecycle**: Pending -> Approved/Rejected
- **Validation rules**: Must record who approved it and when.
- **Risks**: Approval fatigue causing users to blindly approve.

## 14. Artifact
- **Purpose**: Every produced output such as email drafts, reports, PDFs, Word docs, Excel files, markdown, images, drawings, and exported files.
- **Key fields**: `id`, `projectId`, `type`, `uri`
- **Ownership**: Project
- **Relationships**: Produced by an Execution.
- **Lifecycle**: Draft -> Finalized -> Archived
- **Validation rules**: Must be strictly versioned.
- **Risks**: Overwriting user-edited artifacts without merging.

## 15. Knowledge
- **Purpose**: Reusable reference material (e.g., API docs, brand guidelines).
- **Key fields**: `id`, `projectId`, `contentHash`
- **Ownership**: Project
- **Relationships**: Read by Agents as context.
- **Lifecycle**: Ingested -> Embedded -> Archived
- **Validation rules**: Must be verified for accuracy before inclusion.
- **Risks**: Stale knowledge leading to incorrect Agent decisions.
